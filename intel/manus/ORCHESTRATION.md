## Key Architecture Claims (Verified Against Our Scrape)

### Three Layers
1. **Planning & Reasoning** — LLM task decomposition
2. **Action & Execution** — Sandbox, tools, browser
3. **Governance & Validation** — Safety, approval gates, logging

We have all three: planning (plan tool), execution (35 tools), governance (watcher + info loop guard).

### Context Engineering Rules
1. **Stable prefix** — Don't put timestamps at prompt start → We do this (system prompt is static)
2. **Append-only context** — Never rewrite history → Our conversation is append-only
3. **Deterministic serialization** — Stable JSON key ordering → We use sort_keys=True in state.py
4. **Explicit cache breakpoints** — Mark after system prompt → Not implemented (no KV-cache control on llama-server)

### Tool System
- "Never add or remove tools mid-iteration" → We use static tool profiles
- "Consistent naming prefixes" → We have browser_*, file_*, shell_*, message_*
- "Three function-calling modes: Auto, Required, Specified" → We use auto only
- **Logit masking** → This is the key differentiator we're missing. They constrain which tools can be called at each state via token logit manipulation during decoding. We'd need llama.cpp grammar support.

### The Sandbox
- Full Ubuntu 22.04 VM, Python 3.11, Node.js 22, Chromium
- Persistent file system, sudo privileges, internet access
- Task isolation per session
- This matches E2B sandbox service we found in their protos

### Anti-Patterns (Things NOT to Do)
1. Timestamps in prompts → breaks cache (10x cost increase)
2. Dynamic tool swapping → confuses model mid-task
3. Lossy context compression → forgotten context
4. Uniform action patterns → few-shot lock-in
5. No approval gates on high-risk actions

### Implementation Blueprint
They recommend: "Start minimal: build a single agent with 3-5 tools and a simple loop."
We started with 35 tools — maybe too many for smaller models. The tool profile system (core=17, full=35) addresses this.

## What They Built That We Haven't
1. **Logit masking state machine** — constrains tool selection per state
2. **Cloud sandbox (E2B)** — isolated VM per task
3. **Scheduled/recurring tasks** — cron-like agent runs
4. **Slide generation** — HTML→screenshot→PDF/PPTX pipeline
5. **Canvas/image editing** — Upscale, background removal, OCR
6. **Email integration** — Agent receives and reads emails
7. **Enterprise controls** — Data policies, team asset audit logs
8. **Collaboration** — Multi-user session editing with permissions

## What We Built That They Haven't (Open Source)
1. **Local inference** — Runs on consumer hardware (GGUF models)
2. **Ink CLI** — Claude Code-style terminal UI
3. **Project system** — tsunami.md per project (like their knowledge system)
4. **Web builder skill** — Dark-edu template, automated page generation
5. **Zero training required** — Inference only, no RL/SFT pipeline
6. **Completion endpoint** — Bypasses chat template issues entirely

## Notable Quote
> "If model progress is the rising tide, we want Manus to be the boat, not the pillar stuck to the seabed."

Translation: They build the orchestration layer to be model-agnostic. If a better model drops, Manus gets better automatically. Same philosophy as TSUNAMI.
